# SecureFin

SecureFin is an institutional-grade corporate treasury and personal finance portal styled as a zero-trust corporate workstation. It implements real-time client-side Continuous Biometric (Facial) Verification, an automated backend Security Diagnostics Framework, and a Cryptographically-Chained Immutable Transaction Ledger. The application utilizes a full-stack architecture built on React 19, Express, TypeScript, and a Supabase/Firebase relational hybrid backend. It enforces on-device biometric tracking and robust server-side security middleware to mitigate real-world vulnerabilities such as session hijacking, ledger tampering, brute-force attacks, and Cross-Site Scripting (XSS).

## 2. Configuration & Security-Relevant Parameters

SecureFin implements a comprehensive security matrix governing key storage, environment parameters, and secure network transport rules.

### A. Environment Variables (`.env`)
To configure the application, create a `.env` file in the root directory. You can use the following structure as a template (documented in `.env.example`):

```env
# Server Port Mapping
PORT=3000

# JSON Web Token Secret (Used for cryptographic validation of session auth tokens)
JWT_SECRET=your_high_entropy_jwt_secret_key_here

# Local Development Settings
NODE_ENV=development
```

### B. Secure Key & Token Storage
1. **JSON Web Tokens (JWT)**: User authentication tokens generated by the backend are transmitted and stored in the client's local storage under `fintrust_token`. These tokens are dynamically checked against the backend `JWT_SECRET` for active leases.
2. **Biometric Baselines**: When a user registers a face during enrollment, the facial landmark descriptor (a 128-dimensional floating-point array representing spatial facial geometry) is securely stored in the application database. To ensure rapid, low-latency checking, a cached copy is held in the browser's `localStorage` (scoped as `fintrust_face_baseline_<normalized_email>`).
3. **Redaction & Protection**: Plaintext biometric images are **never** stored. Only the cryptographically derived facial descriptor vector is saved, preventing the reconstruction of the user's actual image in the event of a database compromise.

### C. TLS & HTTPS Network Security
- **Secure Transport Enforcement**: The active session validator (`sessionEntropyModule`) continuously checks the transport channel. In staging or production, the application will issue warnings and lock access if a non-HTTPS connection is detected.
- **Client-to-Server Bindings**: The application binds communication exclusively over encrypted tunnels.
- **SameSite and Secure Headers**: All session cookies are configured with `SameSite=Strict` and `Secure` attributes to prevent Cross-Site Request Forgery (CSRF).
- **Network Shield**: The `networkShieldModule` monitors host constraints to prevent DNS rebinding and Man-in-the-Middle (MitM) attacks.

   Launches the bundled production server:
   ```bash

   ```

### C. Run the Automated Security Diagnostic Test Suite
SecureFin comes pre-configured with a powerful automated security testing suite (`verify_security.ts`) that runs integration tests to verify the integrity of the security controls. To execute the test suite:
```bash
npx tsx verify_security.ts
```
The test suite automatically starts a local test server instance and runs the following checks:
1. **Default-Deny Authorization Check**: Confirms that unauthorized requests to protected routes are correctly blocked with HTTP 401.
2. **RBAC Validation**: Verifies that standard users are forbidden (HTTP 403) from accessing admin routes while authorized administrators are allowed.
3. **Input Sanitization Sweep**: Confirms that malicious XSS tags (`<script>`) and SQL Injection payloads are aggressively stripped.
4. **Error Shielding**: Verifies that server crashes are shielded from leaking internal code structures or stack traces to clients.
5. **PII Redaction**: Validates that sensitive credentials (passwords) are redacted within the system audit logs.
6. **Brute-Force Throttling**: Verifies that sending more than 20 login requests triggers a rate limit (HTTP 429).

## 5. Comprehensive Breakdown of Security Features

### 1. Continuous Biometric Verification Model
The centerpiece of SecureFin's defense is the `FaceVerificationProvider` located in `src/FaceVerificationContext.tsx`. It runs a continuous, background surveillance thread:
- **Background Execution**: In the background, the context programmatically spawns a headless webcam stream (320x240) and attaches it to a hidden DOM video element.
- **Warm-Up Buffer**: The stream runs for 2 seconds to allow the camera's auto-exposure and white balance to adjust, guaranteeing high landmarks accuracy.
- **Single-Pass Detection**: `face-api.js` analyzes the frames, detecting facial structures, landmarks, and spatial boundaries.
- **Euclidean Distance Comparison**: The extracted 128-dimensional face vector is compared against the user's registered baseline using a strict Euclidean distance threshold (`< 0.6` implies match).
- **Intrusion Detection**: If multiple faces are detected simultaneously in the frame, the application flags a potential shoulder-surfing threat (`multipleFacesDetected`).
- **Instant Workspace Lock**: If the authorized user leaves the screen, a mismatch occurs, or a multi-face intrusion is flagged, the provider sets `isLocked = true` and injects a full-screen React lock overlay, pausing all workspace functionalities until re-authenticated.

### 2. Default-Deny Access Guard & RBAC
- **Strict Middleware**: The backend auth middleware (`src/middleware/auth.ts`) enforces a default-deny policy. All protected API requests must supply a valid JWT in the `Authorization` header.
- **Role-Based Permissions**: Routes are split into standard and administrative privileges. A user with the role `user` is strictly forbidden from accessing administrative endpoints, which are protected by server-side permission checks.

### 3. Cryptographic Ledger Hash-Chaining
To protect user records from unauthorized database modifications, transactions are structured into a hash-chain:
- Each transaction acts as an immutable block containing its identifier, amount, date, merchant, and the **hash of the previous transaction block**.
- The `ledgerIntegrityModule` traverses this list, recomputing hashes sequentially.
- If a record is tampered with, added, or deleted directly in the database, the cryptographic chain is broken. The system immediately flags an integrity mismatch on the diagnostics dashboard.

### 4. Brute-Force Mitigation (IP Rate-Limiting)
- Enforces an IP-keyed sliding-window rate limiter on sensitive endpoints (e.g., login and registration).
- Limits clients to a maximum of **20 authentication attempts per minute**.
- Excess requests are rejected with `HTTP 429 Too Many Requests`, protecting user accounts from brute-force dictionary attacks.

### 5. Structured Audit Logging
- Enforces system activity logs written locally to `logs/security.log`.
- Redacts all sensitive variables (such as plaintext passwords) to `[REDACTED]` prior to file write.

### 6. Input Sanitization & Error Handling
- Strips harmful HTML, XML, and `<script>` nodes to block Cross-Site Scripting (XSS).
- Sanitizes and escapes input strings to block SQL/NoSQL injections.
- Enforces global error handling, catching all unhandled system failures and mapping them to a generic shielded payload `An unexpected error occurred. Please try again later.`, preventing stack trace information leakage.
